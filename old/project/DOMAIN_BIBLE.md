# DOMAIN_BIBLE

## Глоссарий

| Термин | Значение |
|--------|----------|
| **Meal** | Подтверждённый приём пищи. Immutable после создания. |
| **DraftMeal** | Черновик. Живёт до confirm/cancel. 1 на юзера. |
| **MealComponent** | Один продукт в приёме (курица, рис, масло). |
| **FoodReference** | Запись в справочнике: БЖУ и ккал на 100г. |
| **ParseResult** | Ответ LLM: список компонентов с весами и confidence. |
| **EstimateResult** | Рассчитанные БЖУ/ккал (mean, min, max) — считает ДОМЕН, не LLM. |
| **UserOverride** | Пользовательская правка (вес, БЖУ) через текст. |
| **DayKey** | Строка YYYY-MM-DD в UTC. Граница дня. |
| **Confidence** | LOW / MEDIUM / HIGH — уверенность LLM в распознавании. |
| **Uncertainty Band** | (kcal_max - kcal_min) / 2 — мера неопределённости. |

## Основной цикл (Happy Path)
1. Юзер → фото/текст
2. Создаётся DraftMeal, состояние → DRAFT_MEAL
3. LLM (processDraftConversation) → ParseResult (items + reply)
4. Domain (estimateFromParseWithOverrides) → EstimateResult
5. Бот показывает оценку + клавиатуру [Подтвердить | Отмена | Почему так? | Найдено | Уточнения]
6. Юзер подтверждает → Meal + MealComponents записываются
7. Бот показывает дневную сводку + название блюда

## Правила домена (НЕ НАРУШАТЬ)

### Арифметика
- **LLM НЕ считает финальные ккал/БЖУ** — только domain (`estimation.ts`)
- Формула: protein×4 + fat×9 + (carbs−fiber)×4 + fiber×2 (Modified Atwater)
- Всё хранится как mean/min/max (три значения)
- Если min > mean или mean > max — clamp

### Справочник
- Приоритет при estimation: USDA → FoodAlias → fuzzy (Jaro-Winkler) → OFF text search (макс. 2 продукта, ~7с/шт) → LLM (buildFoodReference) → unknown_generic (0-0-0-0)
- Barcode: OFF API вызывается первым (без rate limit impact), результат персистируется
- LLM-результаты персистируются в FoodReference с verified=false
- Open Food Facts: штрих-коды + bulk-импорт (200+ страниц + 20 категорий при деплое)
- Алиасы: таблица FoodAlias (русские названия → FoodReference), auto-генерируются при OFF импорте
- При матчинге: normalizeName → exact → alias → fuzzy (token coverage + Jaro-Winkler) → fallback
- OFF text search НЕ вызывается при estimation (performance)

### Веса
- Если LLM дал mean/min/max — используем
- Если только mean → min=mean×0.7, max=mean×1.3
- Если ничего → fallback 100г (50-200)
- Юзер может override вес через текст ("рис 200г")

### Подтверждение
- Без confirm → ничего не сохраняется
- Каждый confirm = новый immutable Meal
- Повторные приёмы НЕ дедуплицируются

### Удаление приёмов
- Пользователь может soft-delete приём за сегодня или вчера
- Soft-delete = установить `deletedAt`, запись остаётся в БД
- Удалённые приёмы исключены из: дневных итогов, средних, статистики, истории
- Удаление только через inline-кнопки в «Сегодня»/«Вчера» (не текстовая команда)

### Дневной контекст
- После confirm: дневные итоги (сумма meals за dayKey)
- Статистика: средние за 7/30 дней и всё время
- **Без целей, лимитов, нормативного языка**

## Edge Cases

### Составные блюда (супы, рагу)
- Шире диапазон веса по умолчанию
- LLM разбивает на компоненты

### Текст приоритетнее фото
- Если есть текст + фото, текст главный
- Если юзер говорит "это не X, а Y" — заменяем

### Отрицания
- "убери масло", "не ел хлеб", "без соуса" → убираем из items
- "без кожи", "без костей" → описание приготовления, НЕ удаление

### Уточнения (в режиме CONFIRM)
- Юзер пишет текст → пересчёт через LLM
- Детерминистические правки: "рис 200г", "курица чуть меньше" → без LLM
- Веса продуктов, которые юзер НЕ упоминал → НЕ меняются

### Множественные фото (media group)
- Буферизация 400мс, потом обработка как batch
- Merge: пересечение весовых диапазонов при совпадении продуктов
