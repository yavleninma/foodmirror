### [2026-02] Один монолитный handlers.ts

**Причина:** Быстрая итерация на MVP без абстракций.

### [2026-02] Docker Compose для всего

**Статус:** Действует.

### [2026-02] GitHub Actions для CI/CD

**Статус:** Действует.

### [2026-02-23] Модель LLM только в config, не в .env
**Решение:** Модель OpenAI (gpt-5-mini / gpt-5.2 и т.д.) задаётся в `src/config.ts` (`config.llm.model`). В .env только секреты и идентификаторы (токены, DATABASE_URL, OPENAI_API_KEY, ADMIN_CHAT_ID, FDC_API_KEY).
**Причина:** Минимум параметров в .env; смена модели — правка кода/конфига, не окружения.
**Статус:** Действует. Дефолт по проекту — gpt-5-mini; текущее значение в config — gpt-5.2.

### [2026-02-23] Сообщения коммитов только на английском
**Решение:** Все commit messages пишем на английском.
**Причина:** Кириллица в GitHub Actions / workflow runs отображается сбитой кодировкой (mojibake).
**Статус:** Действует. Правило в AGENT_RULEBOOK.md (стиль коммитов, чеклист).

### [2026-02-23] Conventional Commits (feat, fix, описание, semver)
**Решение:** Формат коммитов — Conventional Commits: `type(scope): description`, типы feat, fix, refactor, docs, test, chore, config и др. Полные правила в `project/COMMIT_CONVENTIONS.md`.

## Данные

### [2026-02] USDA как основной справочник
**Решение:** Импорт USDA FoodData Central (Foundation Foods, SR Legacy).
**Причина:** Открытый, авторитетный источник. ~8000+ записей.
**Статус:** Действует. Обязательный шаг при деплое.

### [2026-02] LLM fallback для неизвестных продуктов
**Решение:** Если продукт не найден в USDA → LLM (buildFoodReference) генерирует приблизительные макросы.
**Причина:** Лучше приблизительная оценка, чем 0-0-0-0.
**Статус:** Улучшено (см. ниже). LLM результаты теперь персистируются.

### [2026-02] Мультисорсный справочник продуктов
**Решение:** Добавлены Open Food Facts (упакованные продукты, штрих-коды) и персистенция LLM-результатов. Цепочка: USDA → FoodAlias → fuzzy (Jaro-Winkler) → OFF API → LLM → unknown_generic.
**Причина:** USDA покрывает только ~8k сырых ингредиентов (US-центричных). OFF даёт 4M+ упакованных продуктов глобально (~30k русских). LLM-результаты теперь сохраняются в FoodReference с verified=false.
**Альтернативы:** FatSecret (бесплатно только US), Nutritionix ($499+/мес) — отложены.
**Статус:** Действует.

### [2026-02] FoodAlias вместо хардкода
**Решение:** Русские алиасы продуктов хранятся в таблице FoodAlias (~229 записей) вместо хардкоженного массива FOOD_NAME_ALIASES в estimation.ts.
**Причина:** Наращивание маппинга без изменения кода, автоматические алиасы при импорте OFF.
**Статус:** Действует.

### [2026-02] FoodReferenceResolver — выделенный сервис
**Решение:** Логика поиска продуктов вынесена из estimation.ts в src/services/foodResolver.ts. In-memory кеш с TTL 5 мин, Jaro-Winkler fuzzy matching, транслитерация.
**Причина:** estimation.ts был перегружен, matching и IO-логика смешивалась с расчётами.
**Статус:** Действует. OFF text search отключён при estimation (performance < 5s), работает только для явных штрих-кодов.

### [2026-02] Immutable Meals
**Решение:** Подтверждённые приёмы пищи нельзя редактировать.
**Причина:** Исторический факт. Corrections = новые записи.
**Статус:** Ключевой принцип. Не менять.

### [2026-02-20] Soft-delete для Meal
**Решение:** Добавлено поле `deletedAt DateTime?` в `Meal`. Пользователь может удалить приём за сегодня/вчера через inline-кнопки в истории. Запись остаётся в БД.
**Причина:** Нужна возможность убрать случайно подтверждённый приём (передумал есть, не доел). Hard delete нарушал бы принцип immutability и ломал аудит-лог.
**Альтернативы:** Hard delete (отклонён), «коррекция записью» (слишком сложно для UX).
**Статус:** Действует. Доступно только для сегодня/вчера.

### [2026-02-20] OFF text search включён в estimateDraft (перед LLM)
**Решение:** Перед вызовом `buildFoodReference` (LLM) для отсутствующих продуктов — пробуем OFF text search для макс. 2 продуктов.
**Причина:** Снижает частоту «выдумка ИИ». Результаты OFF персистируются, следующий запрос мгновенный.
**Альтернативы:** Полностью включить OFF при estimation (отклонено: 6.5с rate limit × N продуктов = неприемлемо).
**Статус:** Действует. Лимит — 2 продукта за запрос.

### [2026-02-20] OFF bulk import: категории + больше страниц
**Решение:** `scripts/import-off.ts` расширен: дефолт 200 страниц (было 50) + 20 категорий OFF (`en:beverages`, `en:coffees`, `en:dairy`, `en:snacks` и т.д.).
**Причина:** Расширение FoodReference без ручных алиасов. Владелец против seed-aliases.ts — только реальные базы.
**Статус:** Действует. Запускается при каждом деплое.

## Релизы и версионность

### [2026-02-23] Semantic Versioning + CHANGELOG + git tags
**Решение:** Версия приложения — в `package.json` (единственный источник). Формат: MAJOR.MINOR.PATCH (semver). История изменений — `CHANGELOG.md` (формат Keep a Changelog). При релизе: обновить version в package.json, добавить запись в CHANGELOG, коммит, затем тег `vX.Y.Z` и push тегов.
**Причина:** Понятно, что задеплоено; можно откатиться к тегу; один источник правды для «что изменилось».
**Статус:** Действует.

### [2026-02-23] Строгое правило: каждый push в main = новая версия + CHANGELOG
**Решение:** CI (GitHub Actions) перед деплоем запускает `scripts/check-release.js`: версия в package.json должна быть строго больше версии из последнего git-тега; в CHANGELOG.md обязательна секция `## [X.Y.Z]` для этой версии. Без выполнения — деплой не запускается.
**Причина:** Исключить попадание в main без версии и описания изменений.
**Статус:** Действует. Локальная проверка: `npm run release:check`.
