**Дата:** 22.02.2026  
**Контекст:** [Аудит продакшена за 7 дней](agent-transcripts) — зафиксированы проблемы с отрицательными БЖУ и игнорированием явных указаний пользователя.

## Проблема 1: Отрицательные и некорректные нулевые значения

**Симптомы:** В БД сохраняются Meal с carbsMean &lt; 0 (пример: Meal 24 «Три куриные ножки» → carbs ≈ -1.35). Возможны аналогичные случаи с protein/fat/kcal.

**Причины (по анализу кода):**
1. `nutrientRanges` в FoodReference (JSON из USDA/OFF) может содержать min/max &lt; 0.
2. `calcTotalsFromComponent` использует `cMin = ranges?.carbs?.min ?? cMean` без проверки на отрицательность.
3. Итоговый `clamp(v) = Math.max(0, v)` в `estimation.ts` уже есть, но он не защищает промежуточные расчёты при нестандартных диапазонах.
4. Возможны ошибки округления и edge cases при scale/пропорциях.

**Стратегия:** защита на нескольких уровнях (defense in depth).

### 1.1 Санитизация nutrientRanges при чтении

**Файл:** `src/domain/estimation.ts`

- В `calcTotalsFromComponent`: для `pMin, pMax, fMin, fMax, cMin, cMax` добавить `Math.max(0, value)` перед использованием.
- Не допускать отрицательных значений в референсных макросах.

### 1.2 Защита в calcTotalsFromComponent

**Файл:** `src/domain/estimation.ts`

- В `calcTotalsFromComponent`: после вычисления `protein`, `fat`, `carbs`, `*Min`, `*Max` — clamp всех значений к `>= 0`.
- Либо ввести внутреннюю `clampNutrient(v)` и применять к каждому выходному значению.

### 1.3 Гарантированный clamp в финальном estimate

**Файл:** `src/domain/estimation.ts`

- Текущий `clamp = (v) => Math.max(0, v)` оставить.
- Добавить проверку на NaN/Infinity и заменять на 0.

### 1.4 Защита при сохранении Meal

**Файл:** `src/fsm/handlers.ts` (функция confirm)

- Перед `db.meal.create`: прогонять `estimate` через `sanitizeEstimate()`, которая clamp’ит все поля к `>= 0` и заменяет NaN/Infinity на 0.
- Это последний рубеж — даже при изменениях в доменной логике данные в БД останутся корректными.

### 1.5 Санитизация reference при resolve

**Файл:** `src/services/foodResolver.ts` (или там, где возвращается reference)

- При возврате FoodReference гарантировать, что `proteinPer100g`, `fatPer100g`, `carbsPer100g` >= 0.
- Для `nutrientRanges` — санитизировать min/max при записи в БД или при чтении.

### 1.6 Тесты

- Unit-тест: `calcTotalsFromComponent` с `nutrientRanges.carbs = { min: -1, max: 1 }` → результат не должен содержать отрицательных значений.
- Unit-тест: `estimateFromParseWithOverrides` с «плохим» reference → итоговый `EstimateResult` без отрицательных и без NaN.

## Проблема 2: Жёсткий приоритет пользовательского текста

**Симптомы:** Пользователь пишет «200 грамм», «60 ккал только углеводы», «30 ккал на 100г» — бот подставляет другие значения или многократно пересчитывает неправильно.

**Принцип:** Текст пользователя = закон. Фото — источник неопределённости. Если пользователь не пишет «не уверен», «примерно» — значения применяются строго.

**Языки:** Целевая модель — мультиязычность (русский пока для теста). В каждом языке множество вариантов формулировок («200 г», «200 грамм», «двести граммов», «около 200г» и т.п.). **Детерминированный парсинг не используем** — он не масштабируется на языки, регулярки ломаются на краевых случаях. Вся интерпретация текста пользователя — через LLM. Существующий `extractWeightOverridesFromText` — не расширять; при возможности со временем отключить или понизить приоритет (LLM — источник истины).

### 2.1 Усиление промптов

**Файл:** `src/config.ts` — `defaultLlmPrompts.draftChat` и `defaultLlmPrompts.parseMeal`

Добавить/усилить:

- «Текст пользователя имеет безусловный приоритет над фото и прошлым контекстом.»
- «Если пользователь пишет число (граммы, ккал, БЖУ) — это точное значение. Не заменяй его своей оценкой.»
- «Для «X ккал»: если указан вес — пересчитывай в ккал/100г; если нет — задай вес, при котором суммарные ккал = X.»
- ««Только углеводы» = protein=0, fat=0, carbs=kcal/4.»

### 2.2 Корректная интерпретация «X ккал» в LLM

Сейчас в промпте: «Если пользователь сказал «60 ккал» — user_kcal_per_100g = 60».  
Для «60 ккал всего» это неверно — 60 это суммарные ккал, а не на 100г.

Уточнить в контракте и промпте:

- `user_kcal_per_100g` — ккал на 100 г.
- Если пользователь сказал «60 ккал» без «на 100г»: считать это суммарными ккал. Тогда нужно задать вес: например, 100 г → 60 ккал/100г, или рассчитать из контекста.

### 2.3 Матчинг overrides к продуктам (LLM → items)

**Файл:** `src/fsm/handlers.ts`, `src/domain/estimation.ts`

Улучшить `findOverrideForItem` и `findOverride`:

- Синонимы: «ягоды» ↔ «berry», «berries», «смесь ягод».
- Добавить в `FOOD_SYNONYMS` частые пары.
- Для фразы «добавь X» — матчить на новый продукт в items; если такого нет, добавлять через LLM или через «виртуальный» item с user override.

### 2.4 Валидация и логирование после LLM

- Логировать случаи, когда в последнем сообщении пользователя явно упоминаются числа (граммы, ккал, БЖУ), но в LLM-ответе соответствующие `user_*` или `weight_g_*` не заполнены — это сигнал к пересмотру промпта или контракта.
- При добавлении новых языков — промпты должны явно перечислять типичные формулировки для каждого языка (или давать LLM инструкцию «принимай любые эквивалентные формулировки на языке пользователя»).

### 2.5 Тесты

- Интеграционные сценарии: пользователь «ягоды 200г 30 ккал/100г» → в итоговом Meal ровно 60 ккал, 0 Б, 0 Ж, 15 У.
- Ручная проверка на разнообразных формулировках (с опечатками, разным порядком слов, синонимами).

## Риски и ограничения

- Усиление промптов может увеличить размер контекста — проверить лимиты токенов.
- Изменения в estimation.ts влияют на все расчёты — обязательны регрессионные тесты.
- LLM может по-прежнему ошибаться на краевых формулировках — логирование и итерация по промптам.
