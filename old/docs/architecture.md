# Архитектура

## Слои

1. Telegram (`src/index.ts`, `src/telegram`)
2. FSM диалога (`src/fsm/handlers.ts`)
3. Domain (`src/domain`)
4. LLM adapter (`src/llm`)
5. Persistence (`prisma`, `src/db.ts`)

## FSM диалога

Состояния (см. `UserState` и `DraftStatus`):

- `IDLE` — нет активного черновика
- `DRAFT_MEAL` — сбор описания/фото
- `ESTIMATED` — получена оценка
- `CLARIFICATION` — уточнение
- `CONFIRM` — ожидание подтверждения

Ключевые переходы:

- Новое сообщение фото/текст → `DRAFT_MEAL`
- Оценка готова → `ESTIMATED` + клавиатура подтверждения
- `Подтвердить` → сохранение `Meal` + возврат в `IDLE`
- `Отмена` → сброс черновика и возврат в `IDLE`
- `Почему так?` → объяснение расчета без изменения состояния

## Поток обработки сообщений

1. При входящем сообщении создается/находится `User`.
2. Входящее сообщение логируется в `MessageEvent`.
3. FSM проверяет команды и контекст:
   - `/start`, кнопки истории, отмена.
4. Для фото/текста создается или обновляется `DraftMeal`.
5. LLM:
   - формирует нейтральный ответ (если нужно),
   - парсит компоненты,
   - возвращает JSON контракт.
6. Domain считает БЖУ/ккал по справочнику `FoodReference`.
7. Пользователь подтверждает — создается `Meal` и `MealComponent`.
8. Бот отправляет сводку дня и название блюда (если сгенерировано).

## Обработка ошибок

- Ошибки LLM и БД пишутся в `logs/error.log` через `logError`.
- Системные сообщения об ошибках отправляются пользователю нейтрально.
- Повторный ввод во время `llmPending` игнорируется до таймаута.
